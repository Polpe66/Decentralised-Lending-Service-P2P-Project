
It's a kind of very, very simple DAO — a Decentralised Autonomous Organisation. For those of you who know what a DAO is, you may have seen this during the theory course, and we mentioned the DAO when we spoke about vulnerabilities, reentrancy attacks, reentrancy vulnerabilities.

How does this work? We have a service that people can go and deposit money into — contribute money — that they can then vote to have lent or borrowed by someone else for a project.

So we have our pool. Think of it as a pool where contributors — the people who can deposit and withdraw money — are depositing value inside. All the value inside here can be borrowed by other users. Each user — contributor 1, contributor 2, contributor 3 — has deposited some amount into the pool. For example: C1 has 3 ETH, C2 has 2 ETH, C3 has 1 ETH. This means the funding pool currently has 6 ETH (3+2+1), which other participants — what we call applicants — can submit proposals to borrow.

Why is it useful to think of the common pool like this? Because as value is borrowed out, that value gets locked. So if we have contributors — contributors may join and leave at any time. For example, C2 decides "I want to withdraw 1 ETH", so now C2 only has 1 ETH available in the pool. Other users may deposit more — C3 may deposit 1 more ETH, or a new user C4 may deposit 1 ETH. So now we have more value inside the pool.

What happens next? The applicants are asking for a loan, and they create a new loan proposal, a new request. This contains some information that we will see: mainly how much money they need, for how long they need it, and what interest they offer — that is, how much will you gain by lending me this money?

So we have, for example, a new proposal: applicant A1 is asking for a loan of 2 ETH.

What happens now? Once the proposal is submitted, a voting period starts. There is some time where all contributors may vote yes or no — approve or deny — on this request. So we may have: C1 votes yes, C3 arrives and votes yes, C4 arrives and votes no, and C2 doesn't care and doesn't vote.

The voting period elapses. So A1, the applicant, can say "I want my proposal to be resolved." We call a function that resolves the voting, and now we start counting the votes. The votes cast by contributors are proportional — weighted — depending on their contribution to the pool. The more money you have deposited, the more important your vote.

This means:
- The "yes" from C1 counts as 3 votes (C1 has 3 ETH disposable).
- The "yes" from C3 counts as 1 vote (C3 has 1 ETH disposable).
- The "no" from C4 counts as 1 no.
- C2, since it did not vote, by default always counts as a reject — so C2 also counts as 1 no.

Overall: 4 yes against 2 no. The vote passes.

We then go and lock some funds. Since 2 ETH was requested, some of the pool funds will be locked. There are specific rules (see the spec) — for this example, assume 2 ETH are now locked inside this loan.

Subsequently, before this loan expires, a new applicant A2 arrives and asks for another 2 ETH. The vote happens again: C1 votes yes, C2 doesn't vote, C3 votes yes, C4 votes no.

Did this proposal pass? No. Now we have a tie. Why? Because C1 is no longer worth 3 but only worth 1. You do not vote with the total amount you contributed to the pool, but with what is called here the available value — the value not currently locked. Since 2 ETH is locked, C1's disposable value is only 1 ETH.

So: C1 = 1 yes, C3 = 1 yes, against C4 = 1 no, C2 (implicit no) = 1 no. A tie. What happens on a tie? As the spec says: the proposal is rejected. A tie counts as a reject.

This is how the dynamic works. Contributors may add or withdraw value at any time. In this case, C1 cannot call "withdraw 1 ETH" because that 1 ETH is locked in a loan — it is not available in the common pool anymore. Until the loan is repaid, that value is unavailable. C4, however, can withdraw 1 ETH because that value is not locked.

What do we mean when a loan gets repaid?

We have the active loan of 2 ETH by A1. A1 can repay this debt. Let's say the interest rate was very high: 50%. The interest is specified as a percentage between 1 and 100 (integer). At 50% on 2 ETH, A1 must repay a total of 3 ETH.

A1 can do partial repayments. So during its life, A1 may repay 1 ETH, then 1 ETH, then another 1 ETH (three installments of 1 ETH = 3 ETH total). Each repayment of 1 ETH is split:
- The base repayment portion refunds the contributors (unlocking their locked value proportionally).
- The interest portion is the gain for contributors — and part of it goes to the compensation pool.

Specifically: each repayment is split into a base component and an interest component. The interest is further split into gain (goes directly to contributors, proportionally by their locked contribution) and collateral (goes to the compensation pool, according to the current collateral percentage). Any leftover due to finite precision arithmetic is credited to the compensation pool.

There are rules for handling precision errors. Consult the spec for the exact formulation.

What happens if a loan is not repaid in time?

If A1 takes 2 ETH and disappears, we have two mechanisms to try to prevent that:

MECHANISM 1 — Bitcoin Liquidity Check

When you create a new loan request, you specify: amount, interest rate, duration, and a Bitcoin address to be used as proof of liquidity. This is described in the spec — it is NOT a real proof of ownership; it is a simplification for didactic purposes only.

What we would ideally want: the applicant proves they own the equivalent in Bitcoin, commits to repay using those Bitcoin funds via some escrow. That is very complex. Instead, the simplified approach is:

The applicant provides a Bitcoin address. At the time of proposal resolution (when the voting period closes), the oracle checks if that Bitcoin address has enough balance to cover the loan (converted using the fixed BTC/ETH exchange rate). If the balance is sufficient, the liquidity check passes. If not, the proposal is rejected — as if a majority reject vote won.

This is clearly insecure: the applicant does not have to own the Bitcoin address, and nothing is locked. It's done purely for the exercise.

How is this possible on-chain? We need an oracle.

THE ORACLE

Think of the oracle as a two-component entity: an off-chain component and an on-chain component.

In our case we use a trusted, centralised oracle:
- Trusted: you trust that the oracle behaves correctly. Whatever it says is true.
- Centralised: you rely on a single entity. Normally oracles are decentralised networks; here it is just one.

On-chain component: A smart contract that stores a map of Bitcoin address → balance (in BTC). This is the data that the lending contract reads. Any smart contract can query this oracle contract and ask "what is the last recorded balance of this Bitcoin address?"

Off-chain component: External code (Python, Java, whatever you prefer) that:
1. Maintains a local, synced version of the Bitcoin blockchain (Bitcoin Core).
2. Keeps an up-to-date UTXO set (so it doesn't have to re-read the entire chain each time).
3. For a given Bitcoin address, sums all UTXOs belonging to that address to compute the current balance.
4. Creates an Ethereum transaction calling the oracle smart contract to update the address → balance mapping.

The oracle needs its own funded Ethereum account to pay gas for these update transactions.

HOW DOES THE ORACLE KNOW WHAT TO UPDATE?

The oracle doesn't update everything constantly (that would be too expensive). Instead:

Before submitting a loan proposal, the applicant sends a transaction to the oracle smart contract requesting a balance update for their Bitcoin address. This request emits an event. The off-chain oracle process listens for these events, wakes up, computes the new balance from the Bitcoin chain, and sends an Ethereum transaction to update the mapping.

To incentivise the oracle to respond, the requester must attach a fee (at least the minimum oracle fee constant, defined as the gas cost of the update operation × 0.1 gwei) when calling the request function.

The fixed BTC/ETH exchange rate is hardcoded: 1 BTC = 30 ETH.

The bitcoin liquidity check at proposal resolution:
- Reads the stored BTC balance for the address specified in the proposal.
- Converts it to ETH using the fixed rate.
- Checks if this ETH-equivalent amount ≥ requested loan amount (and possibly the interest too — this is left as a design decision for you).
- If yes: check passes. If no: proposal rejected.

MECHANISM 2 — Compensation Pool

The compensation pool is a pool of value that accumulates over time from a fraction of each loan's interest. Its purpose is to partially compensate contributors when a loan fails.

When a loan is repaid:
- The base repayment refunds the contributors' locked value proportionally.
- The interest is split into gain and collateral.
  - Gain goes directly to the contributors (not to the funding pool, but to their accounts directly).
  - Collateral goes into the compensation pool.

The split between gain and collateral is governed by the collateral percentage:
- Starts at 50.
- Increases by 5 for every failed loan.
- Decreases by 5 for every successful loan.
- Clamped between 1 and 100.

So if most loans succeed, contributors get almost full interest. If many loans fail, more goes to the compensation pool.

When a contributor asks for compensation for a failed loan (a loan is failed if it was not fully repaid before expiration), they receive from the compensation pool enough ETH to recoup the value they had locked in that loan that was not yet repaid. They receive less than owed if the pool doesn't have enough. The operation can be performed multiple times as the pool refills from other loans. Asking for compensation forfeits proportional applicant repayments for that loan.

IMPLEMENTATION REQUIREMENTS (Section 1.5 of the spec — all mandatory)

SMART CONTRACTS:

1. Events must be emitted as needed — for example when a new proposal is submitted, when a vote is cast, when a loan is created, etc. Emitting events is required. If you skip them, be prepared to defend this strongly at the exam.

2. Each approved loan must be managed by a newly deployed dedicated smart contract (factory pattern). When a proposal passes, you deploy a new loan management contract. The logic for managing repayments, interest distribution, compensation pool contributions, etc., lives in this loan contract, not in the parent contract.

3. Contract termination: properly terminate loan contracts when they are done (fully repaid or failed and compensated). Use self-destruct or appropriate patterns. Do not leave dangling contracts.

4. Main contract upgradability: implement upgradability for the main (parent) service contract. The professor mentioned proxies (as covered two lectures prior). Upgradability is most relevant on the parent contract since it is the factory — changing the loan contract template requires upgrading the parent. Loan contracts themselves do not need full upgradability (though it's your call).

TESTING AND DEPLOYMENT:

5. Use Hardhat for testing. Provide source files to test and showcase correct execution under different circumstances and user behaviours.

6. Use a local private chain based on the provided genesis file. The prefunded accounts from the genesis can ONLY be used to transfer value to other accounts (e.g., fund newly created accounts). They cannot be used to deploy contracts or execute other transaction types.

7. Provide a Python script for the initial service setup (creating accounts, deploying contracts, funding accounts from the genesis prefunded account, deploying the oracle, etc.).

8. Provide a Python script that executes an example set of operations showcasing correct service execution, including printing balances of involved accounts after each meaningful change, and any other relevant state variables.

9. Provide a Python script implementing an automated strategy for a contributor that always votes to approve any new loan proposal (even if the vote is irrelevant because all contributor funds are locked). This requires the contributor to "notice" when new loan proposals are made — use events for this.

10. Provide tests or scripts to measure each operation's gas cost (as seen during lectures). This means programmatic gas measurement, not copy-pasting from Remix.

11. Show how to modify the smart contracts to introduce a reentrancy vulnerability, and provide a test performing a reentrancy attack on the modified code (possibly using an ad-hoc malicious contract).

12. Discuss whether there is a possible malicious strategy for a contributor to gain unfair remuneration at the expense of other honest contributors (without relying on a reentrancy attack), e.g., by leveraging the loan repayment majority rule. Write a script to showcase it if possible.

BITCOIN ORACLE OFF-CHAIN CODE:

The off-chain oracle must be implemented as if new Bitcoin blocks were being generated continuously. The code should process one block at a time. In practice, stop at block 131,000 of the Bitcoin mainnet (same as the lecture examples, to keep file sizes and sync times manageable). However, the code must be correct and would keep working if more blocks were provided — do not pre-process the entire chain and cache results in a way that would break with new blocks.

You may use the BLK files from Bitcoin Core as seen in the lectures. The code should maintain a UTXO set (or similar data structure) to efficiently compute address balances.
